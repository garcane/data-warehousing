2Purpose & Approach

This document is the complete build specification for the SuperstoreDW Power BI report — the reporting layer that sits on top of the Superstore data warehouse. It documents the semantic model (tables, relationships, Power Query, DAX measures) and the report pages exactly as they are defined in the Power BI Project (SuperstoreDW.pbip, TMDL/PBIR format), so that the report can be rebuilt, maintained, or reviewed without opening the project itself.

The report answers the core commercial questions of the Superstore business — how much are we selling, where, to whom, at what margin, and how is that changing over time — for anyone maintaining or extending the warehouse and its reporting layer.

Scope of this version. Version 2.0 extends the as-built documentation of v1.0 with the seven-page target design set out in the supplied Dashboard Guide (DASHBOARD_GUIDE.md, 5 July 2026). Pages 1–2 exist in the project and are documented as built; pages 3–7 are specified in full for construction to be built. Where the guide and the project files differ on page 3, the guide takes precedence (per the sponsor's instruction): the two existing trend charts are absorbed into the new pages 3 and 7 as noted in §9.3 and §9.7. Every planned visual binds only to columns and measures that already exist in the model — the seven-page build requires no new tables, columns, relationships or DAX measures.

Build philosophy. Every table, column, relationship, measure and visual in this specification traces directly to one of four verified sources: the TMDL semantic model definition, the PBIR report definition (report.json), programmatic inspection of the underlying data, or the sponsor-supplied Dashboard Guide. Nothing is assumed or invented. Where the project leaves something unfinished, this is flagged honestly as an open item rather than papered over.

4.3 Relationship register

Five relationships form a clean star: every dimension joins the fact many-to-one, single direction, on a hidden integer surrogate key. Cardinality is not asserted from habit — it is verifiable from the supplied warehouse DDL, where every dimension key is a GENERATED ALWAYS AS IDENTITY PRIMARY KEY (guaranteed unique on the one side) and every fact key column carries a FOREIGN KEY REFERENCES constraint to it.

6Power Query (M) Transformations

There are no transformations in M — all shaping is done upstream in the warehouse SQL, which keeps the model layer thin and the connection repointable by editing one line.

7Calculated Columns

The model defines no calculated columns, and the seven-page build requires none. This is a positive design fact, not an omission: every attribute a visual needs already exists as a physical warehouse column, so nothing is computed in DAX at row level. The date hierarchies used on the trend visuals are Power BI's automatic time-intelligence variations on Date[Date] and Sales[Ship Date], not calculated columns, and require no build action.

8Global DAX Measures

Seven measures are defined, all homed on the Sales fact table, each with a description and format string in the TMDL. All seven are global — no page-specific measures exist, and the full seven-page build of §9 requires no additional measures. Their commercial reading (per the Dashboard Guide):

MeasureBusiness value
Total SalesRevenue health
Total ProfitBottom-line performance
Profit Margin %Efficiency indicator — defined as DIVIDE([Total Profit], [Total Sales]) with a percentage format string (the guide's “× 100” is achieved by the format, not the formula)
Total QuantityVolume performance
Order LinesOperational load
Distinct OrdersCustomer transaction volume
Avg DiscountPricing strategy impact

Total Sales

Total sales amount (additive). Format string: $#,##0. Home table: Sales.

Measure: Total SalesDAX
Total Sales = SUM(Sales[Sales Amount])

Total Quantity

Total units sold (additive). Format string: #,##0. Home table: Sales.

Measure: Total QuantityDAX
Total Quantity = SUM(Sales[Quantity])

Total Profit

Total profit (additive; can be negative). Format string: $#,##0. Home table: Sales.

Measure: Total ProfitDAX
Total Profit = SUM(Sales[Profit])

Profit Margin %

Profit as a percentage of sales. Format string: 0.0%. Home table: Sales.

Measure: Profit Margin %DAX
Profit Margin % = DIVIDE([Total Profit], [Total Sales])

Avg Discount

Average discount rate (non-additive measure). Format string: 0.0%. Home table: Sales.

Measure: Avg DiscountDAX
Avg Discount = AVERAGE(Sales[Discount])

Order Lines

Number of order lines (fact rows). Format string: #,##0. Home table: Sales.

Measure: Order LinesDAX
Order Lines = COUNTROWS(Sales)

Distinct Orders

Distinct order count (uses the degenerate Order ID). Format string: #,##0. Home table: Sales.

Measure: Distinct OrdersDAX
Distinct Orders = DISTINCTCOUNT(Sales[Order ID])
💡 Recommendation — dedicated measures table
House convention prefers a dedicated _Measures table so that a re-import of the fact table can never orphan the measures. The current model homes them on Sales, which works but is more fragile. This is a labelled recommendation only — the specification documents the model as built.

10Cross-Page Requirements

  • Slicers. The only slicers (Geography[Region] basic list; Date[Date] between-range) live on the Executive Summary page and are not synchronised to the other pages — no sync-slicer groups are defined in the report.
  • Filters. There are no report-level, page-level or visual-level filters anywhere in the built definition; the pages to be built add only the two visual-level Top N filters specified in §9.4. The report shows the full 2014–2017 dataset unless a slicer is applied.
  • Theme. The report applies the built-in AccessibleDefault custom theme over base theme CY26SU05; no bespoke theme JSON ships with the project.
  • Canvas. All pages are 1280 × 720 (16:9), layout optimisation None.
⚠️ Open item — global filtering intent
The Dashboard Guide places the Region and Date slicers on the Executive Summary only, and the built report defines no sync groups. With seven pages, unsynchronised slicers mean pages 2–7 always show the full 2014–2017 dataset. If the slicers are meant to control the whole report, enable slicer sync (View → Sync slicers) for both across all pages; this remains a business decision to confirm, recorded here rather than invented.
⚠️ Correction to the Dashboard Guide — cross-filtering scope
The guide states that clicking visuals cross-filters across pages. Power BI cross-highlighting operates within a page only; carrying a selection between pages requires synced slicers or drill-through. The interactive behaviour specified here is therefore: cross-highlighting within each page, plus (pending the decision above) synced slicers for report-wide filtering.
